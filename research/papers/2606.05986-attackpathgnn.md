# Анализ статьи: AttackPathGNN

## Метаданные
- **Название**: AttackPathGNN: Cross-function vulnerability detection in smart contracts using state interference graphs and conjunction pooling
- **Авторы**: Gabriela Dobrița, Simona-Vasilica Oprea, Adela Bâra (Bucharest University of Economic Studies)
- **Год / Venue**: 2026, arXiv 2606.05986
- **Источник**: blockthreat.com newsletter

---

## 1. СУТЬ

GNN-детектор уязвимостей смарт-контрактов, который вместо поиска паттернов внутри одной функции строит граф межфункциональных зависимостей по разделяемому хранилищу (State Interference Graph) и проверяет 8 exploit-предусловий через дифференцируемый AND-агрегатор (conjunction pooling) — коллапсирующий оценку уязвимости если хотя бы одна защита (reentrancy guard, access control, SafeMath) присутствует.

---

## 2. МЕТОД

**Датасеты**:
- **Обучение**: SmartBugs Wild, class-balanced, 3 732 контракта (1866 уязвимых + 1866 безопасных), split 70/10/20 → 2611/374/747
- **Оценка**: SmartBugs Curated, 143 контракта — **все уязвимые**, human-annotated, 10 категорий DASP10

**Важно**: Curated — recall-only бенчмарк. Precision не измерялась.

**6 стадий архитектуры**:
```
Stage 1: AST-парсинг через Slither (fallback: regex)
Stage 2: 30-dim feature vector (13 паттернов + 17 структурных метрик)
Stage 3: State Interference Graph
Stage 4: 8 exploit preconditions
Stage 5: Гетерогенный граф (4 типа узлов, 7 типов рёбер)
Stage 6: GNN (GATv2 + GCN + conjunction pooling)
```

**State Interference Graph** (Stage 3):

Для каждой функции fi: A(fi) = ⟨Ri, Wi, Xi, δi⟩ (reads, writes, external calls, CEI-violation flag).

Два типа рёбер:
```
interferes(fi, fj): (Wi∩Rj) ∪ (Ri∩Wj) ∪ (Wi∩Wj) ≠ ∅
  → три подтипа: write-read, read-write, write-write

can_reenter(fi→fj): |Xi|>0 ∧ pub(fi) ∧ ¬guard(fi) ∧ pub(fj) ∧ (Wi∩(Rj∪Wj)) ≠ ∅
  → направленное ребро reentrancy-пути
```

**8 exploit preconditions** (Stage 4):

| k | Precondition | Смысл |
|---|-------------|-------|
| 1 | externally callable | pub(fi) |
| 2 | has external call | |Xi| ≥ 1 |
| 3 | state after call | CEI violation (запись после внешнего вызова) |
| 4 | no reentrancy guard | guard(fi) = 0 |
| 5 | shares mutable state | кол-во пересечений Wi ∩ Rj с другими публичными функциями |
| 6 | no access control | нет onlyOwner/require(msg.sender) |
| 7 | handles value | payable / msg.value / .balance |
| 8 | unchecked arithmetic | нет SafeMath, Solidity < 0.8 |

**Conjunction pooling**: Φ(fi) = Σ log σ(g(h_c,(i,k) || h_f,i)) — если ЛЮБАЯ защита есть, σ → 0, Φ → -∞. AND-логика встроена в архитектуру.

**Результаты** (5 независимых запусков, seeds 1-5):

| Метрика | Значение |
|---------|---------|
| F1 на SmartBugs Wild test | 92.3 ± 0.2% |
| Detection rate на Curated (weighted) | 90.8 ± 2.5% |
| Detection rate на Curated (macro) | 95.7 ± 0.7% |
| Reentrancy | 98.7 ± 1.8% |
| Access Control | 80.0 ± 5.0% |
| Unchecked Return | 83.1 ± 4.6% |
| 6 категорий DASP10 | 100% на каждом seed |
| Rule-only ablation (без GNN) | 16.8% → +74 п.п. от GNN |

Baselines: Slither v0.11.5, Mythril v0.24.8 (re-executed авторами на том же Curated).

## 3. СКЕПТИК

**Главная проблема: recall-only бенчмарк**

SmartBugs Curated содержит 143 контракта — **все уязвимые**. Precision (false positive rate) не измерялась вообще. Детектор с FP 50% тоже показал бы 90%+ detection rate. Без смешанного датасета (уязвимые + безопасные) в evaluation цифры неполны.

**SmartBugs — старые контракты, НО частично закрыто**

Авторы используют `solc-select per pragma` — компилятор выбирается по версии в контракте, включая 0.8.x. Это снижает проблему из Bastet-статьи (2606.03387). Но основной корпус SmartBugs Wild — исторически v0.4, что создаёт bias в обучении.

**8 preconditions — reentrancy-ориентированы**

Предусловия покрывают классические уязвимости (reentrancy, arithmetic, access control). DeFi-специфика отсутствует:
- AMM price oracle manipulation → нет precondition
- MEV / composability risks (из 2606.05418) → нет
- Flash loan attacks → нет

Это объясняет, почему Access Control на 80% — precondition set просто не захватывает сложные паттерны access control в DeFi.

**Оценивается только detection, не severity**

Выход — бинарный (уязвим / нет). Severity assessment нет. В нашем агенте нужен следующий шаг.

**Воспроизводимость — хорошая**

Multi-seed (5 запусков) с отчётом ±σ — выше стандарта для этой области. Код + hyperparameters опубликованы. Доверие к воспроизводимости высокое.

## 5. РЕШЕНИЕ

**5.1 State Interference Graph → пред-шаг Stage 1**

Перед тем как DiscoveryAgent начнёт анализ, строим явный граф межфункциональных зависимостей по разделяемому storage. Это не ML — это детерминированный анализ через Slither.

```python
# Tool: build_interference_graph(contract_path) → SIG
# Возвращает: пары функций с interferes/can_reenter рёбрами
# Stage 1 использует граф для приоритизации: 
# функции с can_reenter → проверяем первыми
```

Прямая связь с теорией из 2606.05418: `interferes(fi, fj)` ≈ `deps(∆)`, `can_reenter` ≈ конкретизация reentrancy пути.

**5.2 8 preconditions → checklist для Stage 2**

Structured output Stage 2 содержит для каждой находки оценку всех 8 предусловий:

```python
class Finding:
    location: FunctionPair
    preconditions: dict[int, float]  # k: 1-8 → score 0-1
    conjunction_score: float         # Φ(fi)
    mitigations_present: list[str]   # "reentrancy guard", "SafeMath", ...
    severity: Severity               # только если conjunction_score > threshold
```

**Conjunction logic в Guardrails**: если ANY precondition = 0 (mitigation present) → находка маркируется как mitigated, severity снижается. Это же правило, что в architecture AttackPathGNN, но теперь в нашем слое валидации, а не внутри GNN.

**5.3 Slither как primary parser**

AttackPathGNN подтверждает: Slither как AST-парсер с fallback на regex — правильная стратегия. Slither уже популярный инструмент, стоит включить его как обязательный инструмент Stage 1/2.

**5.4 Cross-function scope — явное требование к Stage 1 prompting**

Ключевой тезис статьи: The DAO и Cream Finance — не баги внутри функций, а баги в отношениях между функциями. Stage 1 prompting должен явно требовать:

```
"Analyse pairs of functions sharing mutable storage.
 For each pair: is there a path where fi's external call allows
 re-entry into fj before fi's writes complete?"
```

Без явного указания LLM склонна анализировать функции поодиночке.

**5.5 Что НЕ берём**

Не интегрируем GNN-модель напрямую:
- Узкий фокус (reentrancy-ориентирована, нет DeFi-специфики)
- SmartBugs bias
- Нет precision evaluation

Берём **архитектурные идеи** (SIG, 8 preconditions, conjunction logic), не модель.

## 6. ВОПРОСЫ

- **Precision gap**: нужно оценить FP rate на реальных смешанных контрактах (уязвимые + безопасные). Bastet (2606.03387) — хороший кандидат для проверки.
- **DeFi preconditions**: как расширить 8 preconditions до DeFi-специфики? PATTERN-01 из 2606.05418 (AMM oracle) → precondition 9: «использует внешний AMM для ценообразования».
- **Slither на DeFi v0.8**: насколько Slither стабилен на современных DeFi-контрактах (сложные proxy patterns, nested structs)? Это практический вопрос для нашего инструмента.
- **Связь с Bastet тегами**: 8 preconditions → маппинг на Bastet Tags? `can_reenter` → «Reentrancy», `no access control` → «Access Control», `CEI violation` → под-тег какого тега?
