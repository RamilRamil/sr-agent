# Security Threats and Their Impact on Blockchain Interoperability (2606.14554)

**Source**: https://arxiv.org/abs/2606.14554
**Authors**: Shawn M. Reynolds, Hassan Reza (University of North Dakota)
**Scope**: 38 attacks across 5 categories for interoperable blockchain systems

---

## 5-category threat taxonomy

### A. BC Attacks (12)

| Attack | Impact |
|---|---|
| 51% Attack | Attacker controls consensus, fraud |
| Sybil Attack | Consensus disruption, DoS |
| Double-Spending | Fraudulent spending |
| Block Withholding | Disrupted mining, reward loss |
| Selfish Mining | Unfair rewards, potential fork |
| Fork After Withholding | Forking, reward theft |
| Private Key Leakage | Account takeover, fund loss |
| Nothing at Stake | BC hijack |
| Time-Jacking | Double-spending, resource waste |
| Replay Attack | Transaction duplication |
| Transaction Malleability | Double-spending, fund theft |
| Reentrancy | Financial exploitation |

### B. Network Attacks (7)

DoS/DDoS, Botnet, Packet-Dropping, Stopping Attack, Eclipse Attack, Partitioning, Man-in-the-Middle.

### C. Interoperability-Specific Attacks (6) ← most relevant for bridge audits

| Attack | Real examples | Impact |
|---|---|---|
| Relay Poisoning | ETH Relay | Fraudulent cross-chain txns |
| Oracle Poisoning | single-source oracles | Fund loss, unfair liquidation |
| Goldfinger Attack | new sidechains | Fork, double-spending |
| Sore Loser / Lockup Griefing | atomic swaps | Assets locked indefinitely |
| Unrestricted Deposit Emitting | Multichain, Meter.io | Double-spending without lock |
| Unauthorized Unlocking | Anyswap | Double-spending via stolen keys |

### D. Social Engineering (3)

| Attack | Notes |
|---|---|
| Social Engineering | impersonation, pretexting, whaling |
| Phishing | clone sites, fake ICOs, malicious wallets |
| Homograph Visual Cognitive Deception | Unicode lookalike chars in addresses/code; 50% success on hardware wallets |

### E. Vulnerabilities (10)

Smart Contract Vulns, Public Key Crypto Vulns, Improper Value Validation, Visibility Vulnerability, Weak Randomness, Transaction Order Dependence, Asset Counterfeiting, Verifier's Dilemma, Invalid Consensus Proofs, Inconsistent Event Parsing.

---

## Key countermeasure patterns

- Monitoring/detection systems: address 11+ distinct threats
- Validation techniques: prevent replay and consensus attacks
- Timestamp mechanisms: defend against time-based attacks
- Reputation systems: deter Sybil, eclipse, Goldfinger
- Code audits: prevent smart contract vulnerabilities before deployment

### Selected countermeasures

**Reentrancy**: Check-Effects-Interactions pattern; mutex locks; static/dynamic analysis; fuzz testing.

**Oracle Poisoning**: Aggregate from multiple independent oracles; dispute mechanism; financial incentives for accuracy.

**Unrestricted Deposit Emitting**: Require verified lock events before any deposit; use only secure transfer functions; automated pattern-matching detection.

**Homograph Deception**: Educate users to verify full addresses and checksums; deploy automated monitoring; verify cross-contract function selectors via hex viewing.

**Weak Randomness**: Replace insecure sources (block hash, timestamp) with RANDAO or RBGC.

**Transaction Order Dependence**: Commit-reveal scheme — hide solutions until processed, preventing front-running.
