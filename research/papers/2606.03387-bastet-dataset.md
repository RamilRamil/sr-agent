# Анализ статьи: Bastet Dataset

## Метаданные
- **Название**: Bastet: A Fine-Grained Expert-Labeled Dataset for DeFi Smart Contract Vulnerability Detection
- **Авторы**: Hsu, Wang, Liou, Ke, Toyoda (AIFT, OneSavie)
- **Год / Venue**: 2026, arXiv 2606.03387
- **Источник**: blockthreat.com newsletter

---

## 1. СУТЬ

Датасет из 849 экспертно размеченных находок реальных DeFi-аудитов (Code4rena, 2021–2024) с двухуровневой таксономией уязвимостей (46 тегов + 77 подтегов) — созданный в ответ на то, что все существующие датасеты либо устаревшие, либо синтетические, либо грубо размечены.

---

## 2. МЕТОД

Это **датасет-статья**, не метод. Никаких baseline-сравнений и метрик точности нет — это нормально для такого типа работы.

**Источник данных**: 394 аудиторских отчёта Code4rena, апрель 2021 — ноябрь 2024.  
**Итого находок**: 4 402, из них **849 полностью аннотированы** (19.3%).  
**Аннотаторы**: два независимых эксперта DeFiHackLabs, консенсусный процесс, 100% peer-review.

**Двухуровневая таксономия:**
```
Tag (46)      — root cause механизм  ("Access Control")
  └─ Subtag (77) — конкретный дефект ("Missing onlyOwner modifier")
```
Таксономия не задавалась заранее — **вырастала из данных** в процессе аннотации.

**Распределение (топ-5):**
| Тег | Доля |
|-----|------|
| DoS | 20.7% |
| Input Validation | 13.2% |
| Access Control | 10.0% |
| Logic Error | 8.5% |
| Accounting Error | 8.4% |

182 находки имеют несколько тегов (мульти-label), 139 — несколько подтегов.

**Важный внешний результат, который цитируют авторы**: при переходе с Solidity v0.4 (старые датасеты) на v0.8 (реальный мир) recall у LLM падает до 13% по некоторым типам уязвимостей.

---

## 3. СКЕПТИК

**Что вызывает вопросы:**

- **Только 19.3% размечено.** 849 из 4 402 — небольшая выборка. Могут быть систематические пропуски, если аннотаторы выбирали «интересные» находки.
- **Code4rena — уже отфильтрованные данные.** Судьи Code4rena уже отбрасывают invalid/дубли. То есть датасет смещён в сторону «реальных и интересных» находок — это хорошо для качества, но датасет не репрезентирует распределение уязвимостей в дикой природе.
- **Полный датасет ещё не выпущен.** «Upon conclusion of the competition» — непонятно когда. Сейчас доступны только 849 находок.
- **Лицензия CC BY-NC** — некоммерческое использование. Для обучения модели в коммерческом продукте нужно уточнять.

**Вывод по доверию**: датасет-статьи оцениваются по качеству данных, не по цифрам. Здесь: реальные данные + экспертная разметка + консенсусный процесс = **высокое качество**. Главный риск — малый размер и неполная разметка.

## 5. РЕШЕНИЕ

**5.1 Bastet как эталонный датасет для оценки нашего агента**

Не использовать для обучения (мало данных, лицензия NC), но использовать для **evaluation**: прогнать SR-agent на 849 размеченных находках, сравнить с экспертными тегами. Это честная оценка на реальных Code4rena данных — лучше, чем синтетика.

**5.2 Таксономия (46 тегов + 77 подтегов) → онтология для Stage 1**

Готовая иерархическая структура для классификации находок в Stage 2 (CheckRunner). Вместо того чтобы придумывать свою классификацию — взять Bastet как основу:

```python
# structured output Stage 2
class Finding:
    tag: BastetTag          # из 46 тегов
    subtag: BastetSubtag    # из 77 подтегов
    severity: Severity
    location: CodeLocation
```

**5.3 Распределение уязвимостей → приоритеты Stage 1**

DoS (20.7%) и Input Validation (13.2%) — самые частые в Code4rena DeFi. DiscoveryAgent в Stage 1 должен искать их в первую очередь. Это входной параметр для промпта, не жёсткий код.

**5.4 Recall 13% на v0.8 — критично для нас**

Если мы будем оценивать на старых датасетах (SmartBugs, SolidiFI с v0.4) — получим завышенные цифры, не отражающие реальность. **Тестировать только на v0.8 контрактах** — это наш случай (современные DeFi протоколы).
