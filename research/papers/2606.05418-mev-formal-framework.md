## 1. СУТЬ

Формальная теория безопасности DeFi-композиций: предложены понятия **MEV non-interference** (новый контракт не должен позволять атакующему извлекать больше ценности из жертвы через взаимодействие с внешними контрактами) и **local MEV** (мера потерь жертвы, а не прибыли атакующего), с теоремами о достаточных условиях безопасности и принципами локального рассуждения — позволяющими проверять composability без анализа всего состояния блокчейна.

## 2. МЕТОД

**Тип работы**: формальная, без датасетов и ML-метрик. 18 теорем с доказательствами.

### Два ключевых понятия

**Local MEV** (Definition 3/7):
- `MEV(S, C)` — максимальные потери контрактов C при атаке, ограниченной капиталом атакующего в блокчейне
- `MEV∞(Γ, C)` = max_W MEV(W|Γ, C) — то же, но атакующий может использовать flash loans (неограниченный капитал)

Ключевое отличие от классического MEV [Clockwork Finance]: **local MEV фокусируется на потерях жертвы**, а не на общем MEV всего блокчейна.

**MEV non-interference** (Definition 8):
```
S ̸⇝ ∆  — bounded-wealth: добавление ∆ в S не увеличивает local MEV из ∆
Γ ̸⇝∞ ∆ — unbounded-wealth (flash loans): аналогично
```
Интуиция: взаимодействие атакующего с внешними контрактами в S не должно открывать дополнительные возможности для потерь ∆.

### Достаточные условия безопасности (Theorems 14, 17)

Для bounded-wealth:
1. `MEV(S, C) = 0` — у жертвы нет MEV в изоляции (Airdrop-паттерн), ИЛИ
2. **Observational invariance** + нет токен-потоков из S в ∆

Для unbounded-wealth (flash loans):
1. `MEV∞(Γ, C) = 0` (аналогично), ИЛИ
2. **Observational invariance** (без требования отсутствия токен-потоков — богатый атакующий и так может их создать)

**Observational invariance**: внешние контракты S не могут изменить то, что контракты ∆ наблюдают из S. То есть действия атакующего в S не влияют на вызовы из ∆ в S.

**Sender-agnosticity**: контракт ведёт себя одинаково независимо от того, кто его вызывает.

### Принципы локального рассуждения (Theorems 4, 12, 15, 18)

Можно убрать из анализа контракты, которые **не являются зависимостями** ∆, при условии:
- Граничные контракты (`∂(C,D)`) — sender-agnostic
- Нет токен-потоков из убранных контрактов в ∆ (только для bounded-wealth)

Это позволяет проверять MEV non-interference **локально** — анализируя только ∆ и её прямые зависимости.

### Протоколы из статьи

| Протокол | Безопасен? | Условие |
|----------|-----------|---------|
| Exchange с фиксированным оракулом | ✓ | оракул sender-agnostic, нет токен-потоков |
| Exchange с AMM-оракулом | ✗ | атакующий манипулирует AMM → цена меняется |
| Bank (deposit/withdraw) | ✓ | sender-agnostic |
| FlashLoanArbitrage | ✓ | MEV∞ = 0 (баланс всегда 0) |
| Lending pool | ✓/✗ | зависит от оракула цен |
| Options с AMM-оракулом | ✗ | price manipulation через AMM |

**Ограничения, признанные авторами:**
- **Нет инструмента**: автоматизированная верификация — future work. Сейчас всё вручную.
- **Нет governance/upgradability**: динамические параметры протоколов и DAO-голосования не покрыты.
- **Нет оракулов цен вне блокчейна** (Chainlink): фреймворк моделирует только on-chain взаимодействия.
- **Вычислительная сложность**: проверить MEV non-interference для реального протокола = большая ручная работа.

**Критически важная non-property (Table 3):**
MEV non-interference **не транзитивна**:
```
Γ ̸⇝∞ ∆₁  И  Γ ̸⇝∞ ∆₂  ⟹  НЕ означает Γ ̸⇝∞ ∆₁ | ∆₂
```
Попарная безопасность не гарантирует безопасность совместного деплоя. Это частая ошибка в реальных аудитах.

## 5. РЕШЕНИЕ

**5.1 Composability как отдельный класс проверок в Stage 1**

Stage 1 должна явно различать два вида анализа:
```python
CheckType.STANDALONE      # уязвимости внутри контракта изолированно
CheckType.COMPOSABILITY   # уязвимости через взаимодействие с deps(∆)
```
Для COMPOSABILITY — выявить `deps(∆)` (прямые зависимости) и проверить три условия из Theorem 14/17.

**5.2 Три конкретных паттерна для детекции (Stage 1 prompting)**

Из статьи вытекают три обнаруживаемых паттерна:

```
PATTERN-01: AMM-as-oracle
  Контракт использует AMM для получения цены.
  Риск: атакующий делает большой своп → AMM цена меняется → контракт получает искажённую цену.
  Проверить: есть ли зависимость от AMM.getReserves() или .getAmountOut()?

PATTERN-02: sender-agnosticity violation
  Контракт меняет поведение в зависимости от msg.sender.
  Пример из статьи: require(b || msg.sender == C1) — только C1 может вызвать функцию.
  Риск: атакующий развёртывает контракт-посредник для обхода ограничений.
  Проверить: есть ли require(msg.sender == X) в публичных функциях?

PATTERN-03: non-transitivity trap
  Аудитор проверил A⊕B и B⊕C по отдельности — оба безопасны.
  Вывод «A⊕B⊕C безопасен» — НЕВЕРЕН.
  Stage 1 должна всегда анализировать полную группу зависимостей, не пары.
```

**5.3 Local MEV vs. attacker profit → severity**

Для каждой находки Stage 2 должна считать **потери жертвы**, а не прибыль атакующего:

```python
class Finding:
    local_mev_bounded: Optional[TokenAmount]    # при bounded-wealth атаке
    local_mev_unbounded: Optional[TokenAmount]  # при flash loan атаке
    # severity = f(local_mev_unbounded / contract_tvl)
```

Если `local_mev_unbounded >> local_mev_bounded` — уязвимость требует flash loan, барьер выше.

**5.4 FlashLoanArbitrage — паттерн «нулевого MEV»**

Контракты, чей баланс всегда равен нулю после каждой транзакции (arbitrage, flash loan wrappers), безопасны по Theorem 17 item 1. Это быстрая проверка: если `contract.balance == 0` всегда → MEV∞ = 0 → MEV non-interference выполнена автоматически.

**5.5 Язык для промптинга Stage 1**

Три термина из статьи дают агенту точный язык:
- «Является ли контракт X **sender-agnostic**?» (однозначный вопрос)
- «Каковы **deps(∆)**?» (запрос зависимостей)
- «Есть ли **token flows** из внешних контрактов в ∆?» (конкретная проверка)
