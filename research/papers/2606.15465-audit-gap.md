# The Audit Gap in Blockchain Security (2606.15465)

**Source**: https://arxiv.org/abs/2606.15465
**Author**: Stefan Beyer
**Period**: January 2022 – March 2026
**Data**: 23,818 audit findings (22 firms) + 218 real exploits (~$7.76B losses)

---

## Central finding: two different populations

Code audits and real exploits describe **different phenomena** with non-overlapping distributions.

### What audits find (top categories by frequency)

| Category | % of findings |
|---|---|
| Logic / Business Logic | 14.6% |
| Code Quality | 13.0% |
| Input Validation | 10.0% |
| Access Control | 9.8% |
| Initialization / Upgradeability | 7.3% |
| Integer Overflow | 5.4% |
| Oracle / Price Manipulation | 4.3% |
| **Reentrancy** | **2.1%** |

Critical+High share: stable at 15–17% per year across 4 years.

### What causes real losses (by dollar amount)

| Root cause | Losses | % of total | Avg per incident |
|---|---|---|---|
| Private Key Compromise | $1,894M | 24.4% | $42.1M |
| Phishing / Social Engineering | $1,511M | 19.5% | $125.9M |
| Access Control | $994M | 12.8% | $33.1M |
| Oracle / Price Manipulation | $666M | 8.6% | $15.5M |
| Bridge Exploit | $593M | 7.6% | $148.3M |
| Signature / Replay | $516M | 6.6% | $258.0M |

**Human-vector attacks** (private key, phishing, supply chain, governance):
- 65.6% of annual losses in 2023
- 74.6% in 2024
- 70.8% in 2025
- **49.6% cumulative**

### The gap in numbers

Top-3 audit categories (logic, code quality, input validation) = 37.6% of findings → only **3.8% of losses**.
Human-vector attacks = negligible share of findings → **49.6% of losses**.

---

## Why the gap exists

> "A conventional smart-contract audit reviews the source code of a defined commit against a defined scope. It does not review the deployment environment, signer key-management practices, the CI/CD pipeline, front-end hosting infrastructure, or the supply chain of third-party dependencies."

Code audits can only see what is in the code. Real losses come from:
- Operational security (key theft)
- Social engineering (phishing signers)
- Supply chain (compromised dependencies)
- Governance (on-chain vote manipulation post-deployment)

12 largest incidents at **already audited** protocols: 9 of 12 had root causes outside audit scope.
Example: Bybit ($1.43B) — phishing of signers, contract was audited.

---

## Concentration of losses

- Top 8 incidents = 50.6% of cumulative losses
- Top 20 incidents = 71.4%
- Ethereum + BNB Chain = 89% of incidents, 94% of losses
- Heavy-tailed distribution — Gaussian risk models underestimate tail

By protocol type:

| Type | Incidents | Losses | Avg per incident |
|---|---|---|---|
| CEX | 24 | $2,284M | $95.2M |
| Bridge | 22 | $2,401M | $109.1M |
| Lending | 48 | $736M | $15.3M |
| DEX | 31 | $603M | $19.5M |

Bridges and CEX have highest average loss per incident.

---

## Author's recommendation

> "Audits for the bugs that exist in the code as written, and parallel assessments — of key management, signer workflows, build-pipeline hardening, and dependency supply chain — for the categories that drive the empirically larger share of realised losses."

Code review and operational security are **complementary, not substitutable**.

## Implications for SR-agent

### What this validates

- Our Stage 1–3 pipeline covers code vulnerabilities — necessary but not sufficient
- `BastetTag` 46-tag taxonomy is appropriate for code-layer analysis
- Reentrancy (2.1% of findings) should not dominate Stage 2 attention

### What this suggests adding

**1. Impact map must cover non-code vectors**

`AuditInput.impact_targets` (suggested in pkqs91 notes) should include:
- `loss_of_funds_via_contract` — covered by current pipeline
- `private_key_compromise` — requires key management assessment
- `governance_manipulation` — requires on-chain governance analysis
- `bridge_exploit` — requires cross-chain trust boundary analysis

Different impact targets → different tool prioritization.

**2. Protocol type affects risk profile**

Bridge protocols: avg $148M per incident — highest priority for cross-chain trust boundary checks.
CEX protocols: private key / custody risk > smart contract risk.

Stage 1 SIG prioritization should weight by protocol type, not just code complexity.

**3. Scope declaration in audit report**

Final report should explicitly state what was NOT assessed:
- Key management practices
- Signer workflow security
- CI/CD pipeline integrity
- Front-end hosting
- Dependency supply chain

This matches the paper's finding that most losses come from exactly these out-of-scope areas.

**4. Reentrancy is not the priority it once was**

Only 2.1% of audit findings. Our BastetTag has `reentrancy`, `read_only_reentrancy`, `cross_function_reentrancy` — correct to have them, but Stage 2 time allocation should not overweight them relative to oracle manipulation, access control, and bridge trust boundaries.
