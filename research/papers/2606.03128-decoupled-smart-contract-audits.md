## 3. СКЕПТИК

**Что вызывает вопросы:**

- **Датасет не назван.** 1 899 примеров для detection — откуда? Если синтетические или из одного источника (например, SmartBugs), то train/test leakage очень вероятен и 98.25% нереалистичны.

- **98.25% accuracy подозрительно высока.** Лучшие академические результаты на честных бенчмарках по smart contract auditing — 80–90%. Разрыв в 8+ п.п. над 34B-моделью требует объяснения.

- **"Alignment" 0.4375 — непонятная метрика.** Что именно измеряется? Семантическое сходство с эталонным объяснением? BLEU? Оценка другой моделью? Без расшифровки число бессмысленно.

- **Baseline — только fine-tuned модели.** Нет сравнения с CoT-промптингом больших моделей (GPT-4o, Claude) без fine-tuning — а это реальная альтернатива на практике.

- **Severity dataset несбалансирован**: 549 medium vs 218 low vs 232 high. Это объясняет severity centrality bias у baseline — но нужно смотреть, решает ли их метод проблему реально, а не просто переобучается на этом распределении.

**Вывод по доверию**: числа завышены, но **направление результата** (декомпозиция + маленькие специализированные модели > монолитная большая) — убедительно. SCB — реальная и важная проблема.

## 5. РЕШЕНИЕ

**5.1 Декомпозиция задачи аудита — принять**

Их 4-задачная декомпозиция хорошо отображается на наш pipeline:

```
Stage 1 (DiscoveryAgent) → детекция мест + объяснение
Stage 2 (CheckRunner)    → классификация severity каждого места
Stage 3 (SynthesisAgent) → агрегация + рекомендации по исправлению
```

Это подтверждает нашу иерархическую архитектуру независимым исследованием.

**5.2 Маленькие специализированные модели — исследовать**

Qwen3-0.6B для детекции и Qwen3-4B для объяснения — это GGUF-совместимые размеры. Локальный запуск реален. Но **сначала проверить на своих данных** — их датасет непрозрачен.

Кандидаты для LLM-ядра Stage 1/2: Qwen3-4B, Qwen2.5-Coder-7B (чуть больше, но надёжнее).

**5.3 Chain-of-Verification — взять как паттерн для Stage 3**

Алгоритм для SynthesisAgent при поиске комбинаций уязвимостей:
1. Прочитать результаты Stage 2
2. Сгенерировать верифицирующие вопросы для каждой кандидатной комбинации
3. Ответить независимо
4. Синтезировать только подтверждённые комбинации

Это структурированная альтернатива «попроси LLM найти связи».

**5.4 Severity Centrality Bias — добавить в Guardrails**

Известный failure mode: модели консервативно классифицируют Low и High уязвимости как Medium (~29/30 Low → Medium в baseline). У нас это должно быть явной проверкой в слое валидации:

```python
# Guardrail: если все findings = medium — подозрительно
if all(f.severity == "medium" for f in findings) and len(findings) > 3:
    flag_for_review("severity_centrality_bias_suspected")
```
