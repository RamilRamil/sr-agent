# Оркестратор: исследование и принятые решения

## Что такое оркестратор

Единственная ответственность оркестратора — управлять циклом: дать LLM контекст → получить решение → исполнить → вернуть результат → повторить до завершения. Всё остальное (что решает LLM, как хранится память, что разрешено делать) — ответственность других модулей.

---

## Паттерны цикла

### ReAct (Reasoning + Acting)
Один LLM-вызов чередует мысль и действие. Адаптируется к каждой находке на ходу.
- **Плюсы**: прозрачное рассуждение, гибкость
- **Минусы**: может зациклиться, плохо справляется со сложной декомпозицией
- **Когда использовать**: неизвестно, что найдём (Stage 1, Stage 3)

### Plan-and-Execute
Отдельные проходы: planner строит план, executor исполняет по шагам.
- **Плюсы**: план виден заранее, предсказуем
- **Минусы**: план устаревает на середине без re-planning
- **Когда использовать**: известны шаги заранее (Stage 2)

### Tree of Thoughts / Multi-agent
Параллельные ветки исследуют разные пути, результаты сводятся.
- **Плюсы**: мощно для задач с неизвестным решением
- **Минусы**: дорого, сложно управлять состоянием
- **Когда использовать**: поиск комбинаций / гипотез (расширение Stage 3 в будущем)

## Архитектура для SR-agent: Hierarchical Orchestration

Сценарий использования агента-аудитора задаёт три стадии с разным типом неопределённости:

```
Meta-Orchestrator
│
│  ← управляет потоком (в коде, не в LLM)
│  ← хранит pipeline_state (Orchestration Plane)
│  ← проверяет структуру вывода, а не «намерение»
│  ← считает итерации, ставит hard limit
│
├── Stage 1: DiscoveryAgent (ReAct)
│     Вход: задача + инструменты чтения репо
│     Выход: структурированный список мест → pipeline_state
│
├── Stage 2: CheckRunner (for-loop, детерминированный)
│     Вход: один элемент из pipeline_state за раз
│     LLM вызывается на каждый элемент, но не управляет циклом
│
└── Stage 3: SynthesisAgent (ReAct, широкий контекст)
      Вход: результаты Stage 2 из pipeline_state
      Выход: структурированный отчёт о комбинациях
```

**Ключевое решение**: Stage 2 — детерминированный for-loop без LLM-управления циклом. LLM платим только там, где нужна способность рассуждать в неопределённости.

---

## Два плана: фундаментальное разделение

```
┌─────────────────────────────────────────────┐
│  Orchestration Plane (код)                  │
│  • pipeline_state                           │
│  • счётчики, флаги, маршрутизация           │
│  • whitelist действий                       │
│  • трассировка                              │
│  Недоступен для LLM напрямую               │
├─────────────────────────────────────────────┤
│  LLM Context Plane (промпт)                 │
│  • system prompt                            │
│  • что оркестратор решил показать LLM       │
│  • результаты инструментов                  │
│  ← сюда бьёт Memory Injection               │
└─────────────────────────────────────────────┘
```

**MI может атаковать только LLM Context Plane.** Всё в Orchestration Plane иммунно по конструкции.

**Следствие для pipeline_state**: промежуточные результаты между стадиями хранятся в коде оркестратора (Orchestration Plane), а не в памяти агента. LLM не знает об их существовании и не может их изменить.

## Условия завершения — только в коде

**LLM не решает, когда переходить между стадиями.**

```python
# Плохо — оркестратор читает намерение LLM
if "готово" in output or "I'm done" in output:
    next_stage()

# Хорошо — оркестратор проверяет структуру
output = llm.generate_structured(prompt, schema=StageResult)
# output.status: "in_progress" | "complete" | "needs_input"
if output.status == "complete" and len(output.findings) > 0:
    next_stage()
```

LLM возвращает структурированный вывод (JSON со схемой). MI-payload не может заставить агент досрочно завершить стадию, потому что это условие в коде, не в тексте.

## Observability

### Что логировать (минимум)
На каждое значимое событие — **span** с:
1. Хэш промпта LLM-вызова (не сам промпт — может быть большим)
2. Структурированный вывод LLM (то, на основе чего принято решение)
3. Решение оркестратора (какое правило, почему)
4. Хэш вывода каждого инструмента

### Три роли трассировки

| Роль | Для кого | Содержимое |
|------|----------|-----------|
| Операционный лог | Разработчик | Что, когда, сколько длилось |
| Security audit trail | Расследование MI | Что LLM «решил» и почему |
| Replay record | MI-тестирование (FR-019) | Полный snapshot для воспроизводства |

### Воспроизводимость прогона
Три источника недетерминизма в агенте:
1. **Начальный промпт** — фиксируется как есть
2. **Выводы инструментов** — записываются inline в trace, при replay мокаются
3. **Состояние памяти** — снэпшот до начала прогона

```python
snapshot = {
    "prompt":       "...",
    "memory_state": {records: [...], hashes: [...]},
    "tool_outputs": {"read_file#1": "...", "search#2": "..."},
    "llm_outputs":  [{call_id: 1, response: {...}}, ...]
}
```

**Главный инсайт**: состояние памяти — такая же часть входных данных, как промпт. MI меняет не промпт (на виду), а память (воспринимается как «своя» история агента).

### MI-атака в трассировке
Признак успешной атаки — LLM вернула `findings: []` со ссылкой на «проверено ранее» при непустом файле. Без трассировки виден только пустой отчёт. С трассировкой — момент компрометации.

### Иммутабельность трассировки
Трассировка **append-only**, недоступна для записи из LLM Context Plane. LLM не знает о её существовании. Иначе MI может заставить агента «подчистить» следы.

## Circuit Breakers

### Четыре вида лимитов (все в Orchestration Plane)

```
1. Max iterations  — сколько раз ReAct может позвать LLM
2. Max tool calls  — сколько инструментов за одну стадию
3. Timeout         — сколько времени на стадию целиком
4. Token budget    — сколько токенов потратить на задачу
```

Все четыре — код, не LLM. LLM не знает об их существовании.

### При срабатывании лимита — fail-safe

```python
if iterations >= MAX_ITERATIONS:
    return StageResult(
        status="limit_reached",
        findings=collected_so_far,  # что успели собрать
        reason="max_iterations_exceeded"
    )
```

Никогда не молчать. `limit_reached` попадает в трассировку — человек видит, что стадия не завершилась нормально.

### Ошибка инструмента в Stage 2: retry → skip

Стратегия C: retry N раз, потом явно пропустить.

```python
results[i] = CheckResult(
    status="skipped",
    reason="tool_error after 3 retries",
    findings=None   # None ≠ [] : не проверено ≠ чисто
)
```

`findings=None` vs `findings=[]` — принципиальная разница:
- `[]` = проверили, не нашли
- `None` = не смогли проверить

Stage 3 различает эти случаи при поиске комбинаций.

### Второй уровень: circuit breaker на стадию

```python
error_rate = skipped_count / total_count
if error_rate > 0.5:
    raise StageError("systemic_failure: too many skipped items")
```

Если >50% мест пропущены — системная проблема (нет доступа к репо, неверный путь). Останавливаем Stage 2, сообщаем человеку.

### MI и circuit breakers

Два вектора атаки:
- **«Истощение»** — заставить агента зациклиться и потратить бюджет, не дойдя до реальных уязвимостей
- **«Принудительный пропуск»** — сделать так, чтобы инструмент стабильно падал на нужном файле → агент пропустит его после retry

Защита: все `status="skipped"` выводятся в финальный отчёт отдельной секцией «не проверено». Аудитор видит пропуски явно и может принять решение вручную.

## Фреймворки vs. собственное ядро

### Существующие решения
- **LangGraph** — наиболее релевантен: stateful граф узлов, human-in-the-loop, checkpointing, streaming. Наш Meta-Orchestrator ложится туда почти напрямую.
- **CrewAI** — multi-agent, роле-ориентированный
- **smolagents** (HuggingFace) — минималистичный
- **Pydantic AI** — type-safe, structured outputs по умолчанию

### Проблема для нас
Фреймворки абстрагируют именно то, что нам нужно контролировать — **границу Orchestration Plane / LLM Context Plane**. В LangGraph центральный `state` передаётся между узлами: ничто по умолчанию не мешает LLM-generated контенту оказаться рядом с pipeline_state. Нужно очень намеренно проектировать схему.

Circuit breakers, structured output validation, append-only трассировка — добавляются поверх фреймворка в любом случае.

### Решение
**Написать ядро оркестратора руками (~200-300 строк) → понять фреймворк глубоко → выбрать осознанно.**

Написание руками позволяет почувствовать на практике: где именно проходит граница двух планов, что такое structured output validation «в коде», как circuit breaker встраивается в реальный цикл. После этого LangGraph читается совершенно иначе — видно, что он делает за тебя и что скрывает.
