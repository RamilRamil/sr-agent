# Feature Specification: Ground-Truth Benchmark for Vulnerability Discovery

**Feature Branch**: `023-discovery-benchmark-rig`

**Created**: 2026-07-15

**Status**: Draft

**Input**: User description: "Build the measuring instrument: score any discovery approach against curated ground truth from published audit reports — recall per vulnerability class, precision, and a named list of what we missed."

## User Scenarios & Testing *(mandatory)*

### User Story 1 - See, by class, what our discovery misses (Priority: P1)

A maintainer runs the benchmark over a curated case (a real project plus the findings a published audit reported for it) and gets a scorecard: overall recall and precision, **recall broken down per vulnerability class**, and an explicit, named list of the findings we did NOT surface. For the first time the question "what do we miss?" has a numeric, per-class answer instead of an opinion.

**Why this priority**: This is the entire point. Without it, every future change to discovery is faith, not engineering. The per-class breakdown is what turns "the model misses business logic" from a hunch into a measurement.

**Independent Test**: Point the benchmark at a case with known findings and run the current discovery; the scorecard reports per-class recall, precision, and names every missed finding. Re-running produces the same numbers.

**Acceptance Scenarios**:

1. **Given** a curated case (target + ground-truth findings), **When** the maintainer runs the benchmark with the current discovery, **Then** they get overall recall/precision, per-class recall, and a named false-negative list.
2. **Given** the same case and detector, **When** the benchmark is re-run, **Then** the results are identical (deterministic).
3. **Given** a case whose ground truth contains many classes, **When** the run completes, **Then** EVERY ground-truth finding is scored — none is dropped or deprioritized.
4. **Given** the current discovery scores near zero on business-logic classes, **When** the report is produced, **Then** that is reported plainly as the honest baseline (a useful result, not an error).

---

### User Story 2 - Compare any discovery approach on the same ruler (Priority: P1)

A maintainer plugs in a different discovery approach (today's heuristic, later a taxonomy sweep or a property/fuzzing loop) and scores it on the identical cases and rules, so approaches are directly comparable and progress is visible run over run.

**Why this priority**: A benchmark bolted to today's implementation dies the moment discovery changes. Being approach-agnostic is what makes it a durable instrument rather than a one-off measurement.

**Independent Test**: Register a second (fake) approach and run it over the same case; both produce scorecards computed by the identical rules, and results can be compared side by side.

**Acceptance Scenarios**:

1. **Given** two different discovery approaches, **When** each is run over the same case, **Then** both are scored by the identical matching and metric rules.
2. **Given** a new approach is added, **When** it is registered, **Then** no change to the scoring rules or the dataset is required.

---

### User Story 3 - Honest scoring that cannot inflate itself (Priority: P1)

The maintainer must be able to trust the number. A produced finding only counts as "found" when it structurally corresponds to a ground-truth finding — same place, same class. Anything vague, partial, or arguable is reported as needs-review and never silently credited. Ground truth itself comes from a published human audit, not from a model.

**Why this priority**: This is the project's hardest-won lesson applied to measurement: a permissive gate ("it compiled") once made worthless proofs look successful. A permissive matcher would make worthless discovery look successful — and we would optimize toward the lie.

**Independent Test**: Feed the scorer a near-miss (right place, wrong class; or right class, wrong place) — it is NOT credited as found; it appears as needs-review/unmatched. Recall never rises from ambiguity.

**Acceptance Scenarios**:

1. **Given** a produced finding at the same place AND of the same class as a ground-truth finding, **When** scored, **Then** it counts as found.
2. **Given** a produced finding matching only on place, or only on class, or by loose textual similarity, **When** scored, **Then** it is NOT counted as found; it is reported as unmatched/needs-review.
3. **Given** produced findings with no ground-truth counterpart, **When** scored, **Then** they are reported as false positives (they lower precision, never raise recall).
4. **Given** ground truth, **When** it is loaded, **Then** it originates from a curated, human-verified label file — never generated by a model at run time.

---

### Edge Cases

- A label file names a vulnerability class outside the accepted taxonomy → loading fails loudly (not a silent skip that would quietly shrink the denominator).
- The dataset root or a case path points inside the agent's own project area → rejected (target/finding material must stay external).
- A detector returns nothing → recall 0, every ground-truth finding listed as missed; no crash.
- A detector returns hundreds of guesses → precision collapses; recall cannot rise from volume alone (only structural matches count).
- A ground-truth finding spans multiple places → matching is defined and documented; ambiguity resolves to needs-review, never auto-credit.
- Two produced findings match one ground-truth finding → counted once; the surplus is a false positive.

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: The system MUST load benchmark cases from a dataset root configured outside the agent's own project area; the root and case paths MUST be validated as external, and no dataset content may be committed with the agent.
- **FR-002**: A case MUST describe its target (a location or repository reference) and a curated ground-truth finding list, each finding carrying an identifier, a vulnerability class, a location, a function name, and a severity.
- **FR-003**: Ground truth MUST be human-curated/verified (transcribed from a published audit), never produced by a model during a run.
- **FR-004**: Loading MUST reject a finding whose vulnerability class is not in the accepted taxonomy — loudly, never as a silent skip.
- **FR-005**: The system MUST support pluggable discovery approaches: an approach receives a case and returns candidate findings in the same shape; adding one MUST NOT require changing the scoring rules or dataset.
- **FR-006**: A produced finding MUST count as found ONLY when it structurally corresponds to a ground-truth finding on BOTH location and vulnerability class; partial, loose, or textual-similarity matches MUST NOT be credited.
- **FR-007**: Non-credited correspondences MUST be reported explicitly as unmatched/needs-review; ambiguity MUST NEVER increase recall.
- **FR-008**: The system MUST report overall recall and precision, **recall per vulnerability class**, and an explicit named list of ground-truth findings that were missed.
- **FR-009**: EVERY ground-truth finding in a case MUST be scored — the system MUST NOT prefilter, sample, or deprioritize any finding.
- **FR-010**: Results MUST be reproducible: the same case + approach produces identical numbers across runs.
- **FR-011**: Results MUST be emitted both human-readably and machine-readably, written outside the agent's project area (never committed).
- **FR-012**: The scoring itself MUST require no model and no paid service; it MUST run offline.
- **FR-013**: The system MUST ship a baseline approach wrapping the current discovery so the first run yields an honest starting number, reported plainly even when it is near zero.
- **FR-014**: The behavior MUST be validated by offline, deterministic tests over a small SYNTHETIC fixture dataset (invented, not real audited material): external-root and taxonomy validation, the anti-inflation matching property (near-misses not credited), per-class recall arithmetic, and an end-to-end scoring run with a fake approach.
- **FR-015**: Documentation MUST record the case layout, how to add an approach, how to read the scorecard, and the honesty rules (structural matching, no prefiltering, human-curated ground truth).

### Key Entities *(include if feature involves data)*

- **Benchmark Case**: one curated unit — a target reference plus its human-curated ground-truth findings; lives outside the agent's project area.
- **Ground-Truth Finding**: a published audit's finding, transcribed with identifier, vulnerability class, location, function, severity. The denominator of recall.
- **Discovery Approach**: a pluggable producer of candidate findings for a case; the thing under measurement.
- **Match Outcome**: per produced/ground-truth pair — found, missed (false negative), spurious (false positive), or needs-review (never credited).
- **Scorecard**: overall recall/precision, per-class recall, the named missed list; human- and machine-readable, written outside the agent's project area.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: For a curated case, the maintainer obtains per-class recall, precision, and a named list of every missed finding in a single run.
- **SC-002**: Re-running the same case + approach yields identical results 100% of the time.
- **SC-003**: 100% of ground-truth findings in a case are scored (no prefiltering/sampling).
- **SC-004**: In 100% of defined near-miss tests (right place/wrong class, right class/wrong place, textual-similarity-only), the match is NOT credited to recall.
- **SC-005**: No dataset content (targets, reports, findings, paths) is committed with the agent, and scoring runs fully offline with no model or paid service.
- **SC-006**: A first baseline number exists for the current discovery, reported honestly per class — including where it is zero.

## Assumptions

- The first curated case comes from an audit report the operator already holds; transcribing its findings into labels is operator work (the only judgment call is assigning each finding a taxonomy class).
- The accepted taxonomy is the project's existing vulnerability-class list; classes outside it are a load error rather than a new category invented at run time.
- Location matching normalizes to contract/file + function; a documented, deterministic rule decides correspondence, and anything it cannot decide becomes needs-review.
- The benchmark measures DISCOVERY (did we surface the finding), not proof quality — proof quality is already gated by the existing proof pipeline.
- A single case is enough to start; the instrument is designed so more cases are added without code change.

## Out of Scope

- Building a better discovery approach (taxonomy sweep, property/fuzzing loop) — this instrument exists to measure that next work.
- Changing the current discovery stages or the proof pipeline.
- Committing any real audited target, report, or finding data with the agent.
- A model-as-judge matcher (deliberately rejected: it would measure a model against a model).
- Scoring proof/PoC quality, or any user interface.
