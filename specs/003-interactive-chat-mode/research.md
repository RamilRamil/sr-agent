# Phase 0 Research: Interactive Chat Mode

Every question here was resolved by reading the existing codebase, not by picking a library. The central finding: the hard part of this feature — a decision loop that validates actions, gates irreversible ones out-of-band, and wraps all external data — **already exists** (`orchestrator/loop.py` + `orchestrator/action.py` + `orchestrator/confirmation.py` + `orchestrator/context.py`). It has zero call sites anywhere in the codebase (confirmed by `grep -rn "orchestrator.loop\|orchestrator\.action\|orchestrator\.context\|orchestrator\.checkpoint"` across `sr_agent/` and `scripts/` — no hits outside the modules' own definitions). `guardrails/escalation.py::evaluate_triggers` is in the same state — built, tested in isolation presumably, never invoked. Research here is mostly "how do we wire this in correctly and what has to change," not "what do we build from scratch."

## R1 — What is "the existing AgentAction decision loop" the spec refers to?

**Decision**: It's `orchestrator/loop.py::OrchestratorLoop`. Reuse it directly; do not write a parallel loop.

**Rationale**: `OrchestratorLoop.run()` already does, per-iteration: build DATA-wrapped messages (`context.build_messages`/`wrap_data`) → call the LLM, parse strictly as `AgentAction` (`llm_core/schemas.py`) → persist any reported finding → check terminal actions (`escalate`, `complete`) → `validate_action()` (whitelist + param validation + reversibility annotation) → if the action requires out-of-band confirmation (`REQUIRES_OOB_CONFIRMATION = {write_poc, run_tests, deploy_test_contract}`), write a pending request and block on `check_confirmation()` (fail-safe: timeout → rejected) → only then dispatch. This is exactly FR-002 (single decision structure), FR-004 (DATA-wrapping on every turn), FR-005 (hard OOB gate), and most of FR-006 (bounded iteration) already implemented and, per the code's own docstring, already the intended mechanism ("Main ReAct loop for Stage 1 and Stage 3").

**Alternatives considered**: Writing a new chat-specific loop from `AgentAction`/`validate_action` primitives directly. Rejected — it would duplicate `loop.py` almost line for line while leaving the original dead code in the tree, and FR-002 explicitly requires reusing the existing decision structure, not inventing a second one.

## R2 — `OrchestratorLoop` calls `ClaudeClient()` (paid Anthropic API). The project avoids paid-API dependency. How does chat mode use this loop without that dependency?

**Decision**: Introduce `llm_core/chat_reasoning.py::ChatReasoningProvider`, matching the narrow interface `OrchestratorLoop` actually needs from `self._llm` (`.complete(messages) -> AgentAction`), and inject it into `OrchestratorLoop` instead of `ClaudeClient()`. Internally: if `LocalClient.available()`, call the local model with a chat-adapted system prompt (same `AgentAction` JSON contract `ClaudeClient._SYSTEM_PROMPT` already documents, reused verbatim for the schema portion) and parse the response the same way `ClaudeClient._parse_response` does (strict JSON, no execution of the text). If the local model is unavailable, **do not** fall back to relay — return a sentinel the loop turns into "turn blocked, local model unavailable" per FR-011 (user-confirmed: refuse and wait, never silently escalate). Escalation to the relay only happens on a **deterministic trigger** (see R5), using the same pause/resume file-based mechanism `orchestrator/relay.py` + `planner/stage2.py::run_stage2` already use for Stage 2 — a chat turn that escalates writes a relay request and returns a "paused, pending relay response" status rather than blocking the terminal.

**Rationale**: `OrchestratorLoop` was written against a narrow, swappable interface almost by accident — it only ever calls `self._llm.complete(messages)`. Nothing else in the loop is Claude-specific. This makes the injection point clean: `OrchestratorLoop.__init__` needs one new parameter (an injectable reasoning provider, defaulting to preserve existing non-chat callers if any appear later) rather than a rewrite.

**Alternatives considered**: Modifying `ClaudeClient` itself to add a local/relay mode. Rejected — `ClaudeClient` is a thin, correct wrapper around the real Anthropic SDK with extended-thinking always on; conflating it with the local/relay routing would make an unrelated paid-API client responsible for logic it has nothing to do with, and every other part of the system (Stage 2) already keeps "which backend" as a separate concern from "the client for that backend."

## R3 — What counts as "escalate to the stronger reasoning path" — a fuzzy complexity judgment, or something concrete?

**Decision**: Concrete and closed. Two sources, both already built: (a) the model's own `AgentAction.escalation_trigger` field (it can self-report), and (b) `guardrails/escalation.py::evaluate_triggers()`, a deterministic, non-LLM function checking the 8 `EscalationTrigger` cases (irreversible action requested, memory status change from a non-human source, critical finding, unverified high, mock-detected PoC, contradicting findings, unknown pattern / missing Bastet tag, resource limit approaching). `evaluate_triggers` runs on every turn's action/finding regardless of what the model said, so an evasive or manipulated model response can't suppress escalation by simply omitting `escalation_trigger` — this is the direct mechanism behind FR-004's "regardless of phrasing."

**Rationale**: The user's original framing ("local model for simple/cheap routing, escalate to relay/Opus for hard reasoning") sounds like it wants a complexity heuristic. There already is no such heuristic in the codebase, and inventing one (e.g., an LLM-judged "is this hard" classifier) would itself be an unverifiable, promptable decision point — exactly the kind of soft judgment call this project's whole design tries to avoid making load-bearing. The existing 8-trigger set is deterministic, testable, and already covers the security-relevant cases (irreversible action, status change, critical/unverified findings). "Hard reasoning" in practice becomes "the deterministic guardrail fired" — which is a better definition than a vibes-based one anyway.

**Alternatives considered**: A token-count or prompt-complexity heuristic for escalation. Rejected — not derived from anything in the existing system, adds a new untested decision surface, and doesn't map to any concrete requirement in the spec (FR-002 says "escalate... using the same decision structure already used for audit-stage analysis" — the decision structure already used is exactly the 8 triggers plus the model's self-report, nothing else).

## R4 — `MAX_ITERATIONS = 50` in `loop.py` is a whole-session cap. FR-006 wants a per-turn budget. How do these relate?

**Decision**: Reframe as **per-turn**, not whole-session. Each user message resets a fresh iteration counter with the existing `MAX_ITERATIONS`-style cap (exact number is a tuning constant, not a new mechanism); the chat *session* itself can span an unbounded number of turns (that's the whole point of a REPL), but any single turn's internal tool-call loop stops at the budget and reports state honestly, per FR-006.

**Rationale**: `loop.py`'s current 50-iteration cap makes sense for a single, bounded audit-stage run (Stage 1 or Stage 3, one shot, then done). A chat session is not one bounded run — it's a sequence of many small bounded runs (turns). Reusing the same loop body per-turn, with the counter reset at the top of each turn instead of once per session, satisfies FR-006 without inventing a new stopping condition.

**Alternatives considered**: One session-wide budget shared across all turns. Rejected — it would mean a long, productive session eventually locks up permanently regardless of how well-behaved individual turns are, which isn't what FR-006 is protecting against (FR-006 is about *one bad turn*, not aggregate session length).

## R5 — How do chat sessions persist and resume across separate CLI invocations (FR-012, user-confirmed)?

**Decision**: A `ChatSession` (new, `models/chat.py`) persisted via `EpisodicMemory`, following the exact pattern `orchestrator/checkpoint.py` already uses for audit-session checkpoints: each turn is written as a `MemoryRecord` with `target=f"chat:{session_id}"`, `source_type` set per the record's actual origin (see R7), and `session_id` set to the chat session's id. Resuming means loading all records for that target key via `EpisodicMemory.load(project_id, target)` and reconstructing the session's grounding facts and turn history from them, the same way `checkpoint.load_checkpoint` reconstructs a `Checkpoint` from the max-timestamp record.

**Rationale**: `EpisodicMemory` already provides append-only, HMAC-signed, principal-scoped persistence, and `checkpoint.py` already demonstrates the exact "session-scoped target key, one record type layered inside `MemoryRecord`" pattern this needs. No new storage mechanism, no new file format, no new integrity story to build or audit — the HMAC signing and silent-drop-on-tamper behavior chat sessions get is identical to everything else in memory, for free.

**Alternatives considered**: A separate SQLite/file-per-session store outside `EpisodicMemory`. Rejected — it would need its own integrity story (tamper detection, principal scoping) duplicating what `EpisodicMemory` already does correctly, for no benefit; chat turns are exactly the same shape of "append-only record with a source-type trust tier" that memory already models.

## R6 — How does a session stay consistent with facts established many turns ago once the raw conversation exceeds the local model's ~28K-token window (FR-009, SC-004)?

**Decision**: Maintain a small, structured **session-facts** record (bound `project_id`, referenced `finding_id`s, a short list of recent tool-result summaries) that is updated deterministically by the orchestrator (not by the model) whenever a finding is persisted or a notable tool result lands — the same trust posture as a checkpoint (`tool_output`/orchestrator-authored, not LLM-inferred). Extend `context.py::build_messages`'s existing truncation-priority list (today: knowledge_chunks dropped first, then older tool outputs, system prompt never truncated) with the session-facts bucket at the *top* of that priority order — included on every turn, dropped last if ever.

**Rationale**: `build_messages` already implements exactly this kind of priority-ordered truncation for a different input (knowledge chunks vs. tool output); adding one more priority tier is a small, consistent extension, not new infrastructure. Because the facts are orchestrator-maintained rather than summarized by the (possibly resource-constrained, possibly adversarial-input-exposed) model itself, a confused or manipulated local-model turn can't corrupt what the session "remembers" about its own project scope — it can only fail to answer well about it, which is a quality problem, not a trust problem.

**Alternatives considered**: Model-driven summarization (periodically asking the local model to summarize the conversation so far). Rejected — it reintroduces exactly the trust problem the rest of this project works hard to avoid: a summary is model output, and if it's later fed back in as "established fact," an adversarial or malfunctioning turn could poison what the session believes about itself. Deterministic, orchestrator-authored facts avoid that entirely.

## R7 — `OrchestratorLoop._persist_finding` writes chat/audit findings with `source_type=SourceType.llm_inference`. Is that correct for chat?

**Decision**: No — fix it to `SourceType.external_llm_output` for anything produced by the reasoning provider (local model or relay), matching what `planner/stage2.py` already does for both `run_stage2` (relay) and `run_stage2_local` (local model): both set `source_type=SourceType.external_llm_output` with the rationale "automation != authoring." `loop.py` predates that convention (or was simply never reconciled with it, being orphaned) and uses the weaker `llm_inference` tier instead.

**Rationale**: Per the project's trust hierarchy (`human_input` > `tool_output` > `external_llm_output` > `llm_inference`), `llm_inference` is the *lowest* tier — reserved for the model reasoning without any external grounding at all. A chat turn's finding, produced by either the local model or the relay, is exactly the same kind of output Stage 2 already classifies as `external_llm_output`; using the weaker tier for chat would be an inconsistency with no justification, and would matter concretely for `evaluate_triggers`' check #2 (memory status change), which treats `source_type != human_input` as escalation-worthy regardless of which of the two non-human tiers it is — but getting the tier right still matters for every other place in the codebase that reasons about provenance (e.g., `for_llm_context()` and reporting).

**Alternatives considered**: Leaving it as `llm_inference` and treating the mismatch as out of scope. Rejected — this plan reuses `loop.py`; leaving a known, easily-fixed trust-tier bug in place while wiring the module up for the first time is not a reasonable simplification, it's importing a latent bug into the live path.

## R8 — What does FR-008/FR-013's "lightweight visibility, not a gate" apply to, given every `write_execute` action in the current taxonomy is already marked irreversible (hard-gated by FR-005)?

**Decision**: FR-008/FR-013 is a **notice that a hard OOB confirmation is being requested**, shown immediately in the chat as the request is filed — not a second, softer gate that could be satisfied instead of the hard one. Concretely: when a turn's action requires OOB confirmation, the agent's chat-visible response for that turn is "requesting confirmation to `write_poc` for finding X (confirmation id `<id>`) — run `sr-agent confirm <id> --approve` in another terminal" and the turn returns *immediately* with a "paused, pending confirmation" status rather than blocking the process on `check_confirmation()`'s poll loop (which defaults to a 300s timeout). The user resumes the session (R5) once they've confirmed out-of-band, at which point the turn's dispatch completes and reports pass/fail.

**Rationale**: All three `write_execute` actions (`write_poc`, `run_tests`, `deploy_test_contract`) are marked `REVERSIBLE[...] = False` in `models/action.py` today, and the spec's own Assumptions explicitly forbid this feature from reclassifying anything ("does not change... how any existing action is classified as irreversible"). So there is no existing action type that is genuinely "consequential but not irreversible" — the FR-008 spec text's illustrative example (writing a PoC file) turns out, on reading the actual registry, to already be hard-gated by FR-005. The reading above is the only one consistent with both requirements simultaneously: FR-008/013 governs *how the hard gate is surfaced in conversation* (immediately, without an extra soft confirmation step of its own), not a separate, weaker gate that could let a write-execute action through without OOB approval.

**Alternatives considered**: (a) Reclassify `write_poc` as reversible so it only needs the lightweight notice. Rejected outright — directly contradicts the spec's own Assumptions and would repeat exactly the shortcut `scripts/poc_queue_runner.py` took (and explicitly flagged as not to be copied into the real implementation) of writing a test file into a real target repo without out-of-band human sign-off. (b) Block the chat process for up to `confirmation_timeout_s` (300s) waiting on `check_confirmation()`, as `loop.py` does today for its batch-run context. Rejected for interactive use — freezing a live REPL for up to five minutes is a worse experience than pausing and letting the user resume, and R5 already gives the session a resumable shape for exactly this reason.

## R9 — Does `write_execute` dispatch (write_poc/run_tests) need new code, or does it already exist?

**Decision**: The tool functions already exist and are already correct (`tools/write_execute.py::write_poc/run_tests`, `tools/sandbox.py::DockerSandbox`) and are already registered in `tools/registry.py` with the right `write_execute` classification and confirmation-required description. What's missing is purely the dispatch wiring: `OrchestratorLoop._dispatch()` currently returns `"[STUB] Tool {action_type} not yet implemented."` for every action type except `read_file`/`search_code`/`analyze_transactions`. This plan implements the `write_poc`/`run_tests` branches in `_dispatch()` to call the existing tool functions (post-confirmation, as the loop already guarantees by the time `_dispatch` is reached for a gated action).

**Rationale**: No new tool logic needed — this is the same gap the rest of the plan keeps finding: correct, tested building blocks that were never connected to the loop that's supposed to call them.

**Alternatives considered**: None — this one wasn't a design choice, just confirming there's no missing primitive.
